# 하네스 엔지니어링 (Harness Engineering) — 프로젝트 지침

이 문서는 Claude Code가 이 프로젝트에서 작업할 때 따라야 할 규율(harness)을 정의합니다.
"프롬프트 한 줄"이 아니라, **코드 · 스크립트 · 참고 문서 · 체크리스트**로 AI의 행동 범위와 절차를 강제하는 것이 핵심입니다.

## 핵심 원칙

1. **실행이 아니라 설계가 본질이다.**
   사람이 할 일은 작업 단계를 정밀하게 쪼개고, 각 단계의 입력/출력/검증 기준을 정의하는 것.
   Claude Code는 그 설계를 실행하는 엔진.

2. **모든 작업은 3단계로 분리한다.**
   - `INPUT`: 무엇을 받는가 (원본 자료, URL, 파일, 이전 단계 산출물)
   - `PROCESS`: 어떤 단계를 거치는가 (아래 workflows/ 참고)
   - `VALIDATE`: 완료 기준이 무엇인가 (체크리스트, 스스로 채점)

3. **신입 사원 OJT처럼 다룬다.**
   - 처음 결과물은 반드시 사람이 검토
   - 반복되는 지적 사항은 이 CLAUDE.md 또는 workflows/*.md 에 규칙으로 추가
   - 규칙이 쌓일수록 사람의 개입은 줄어든다 (Auto-Research 루프)

4. **벤치마킹을 적극 활용한다.**
   구현 전에 참고할 만한 기존 사례(GitHub 리포, 유사 도구)를 먼저 찾고,
   "이런 방식으로 만들어줘"라고 구체적으로 요청한다. 비개발자도 이 방식으로 전문가 수준 도구를 만들 수 있다.

## 작업 시작 전 체크리스트

- [ ] 이 작업에 해당하는 workflow 파일이 `workflows/`에 있는가? 없으면 먼저 만든다.
- [ ] INPUT의 출처와 형식이 명확한가?
- [ ] VALIDATE 기준(완료 조건)이 workflow에 이미 적혀 있는가?
- [ ] 참고할 만한 기존 사례/템플릿이 있는가? (있으면 scripts/ 나 workflows/에 링크 남기기)

## 디렉토리 구조
## 새 workflow 추가하는 법

1. `workflows/` 에 `업무명.md` 파일 생성
2. INPUT / PROCESS / VALIDATE 3섹션으로 작성
3. 처음 몇 번은 사람이 결과 검토 → 반복 지적사항을 PROCESS나 VALIDATE에 규칙으로 반영
4. 안정화되면 scripts/에 자동화 스크립트로 격상 고려

## 주의사항

- 반복 업무라도 "완전 자동 실행"보다 먼저 "사람 검토 후 승인" 단계를 거쳐 안정성을 확인한 뒤 자동화 수준을 높인다.
- 개인정보(고객 이메일, 세금계산서 등)를 다루는 workflow는 scripts/에 별도로 접근 권한/로그 규칙을 명시한다.

## 팀장 에이전트 운영 방식

**사용자는 이 메인 세션(팀장)에게만 지시한다.** 팀장은 직접 모든 작업을 처리하지 않고,
아래 서브에이전트(`.claude/agents/`)에게 위임한 뒤 결과를 종합해서 사람에게 보고한다.
서브에이전트는 Claude Code가 각 파일의 `description`을 보고 자동으로 판단해 호출한다 —
사람이 "content-creator 써줘"라고 지목할 필요 없이, 팀장이 알아서 라우팅한다.

### 서브에이전트 라우팅 표

| 서브에이전트 | 담당 | 언제 위임하는가 |
|---|---|---|
| `content-creator` | 교육설계, PPT, 블로그, 유튜브 스크립트 | "만들어줘/써줘" 류의 콘텐츠 제작 요청 |
| `researcher` | 뉴스 모니터링, 이메일 분류, 크롤링 | 정보 수집·정리·요약 요청 |
| `sales-ops` | 고객 응대, 견적서, 팔로업 | 리드/고객 커뮤니케이션 관련 요청 |
| `daily-assistant` | 캘린더, KTX/SRT 좌석 모니터링, 카톡 아카이빙 | 일상적인 개인 비서 요청 |
| `harness-designer` | 신규 반복업무의 workflow 설계 | 위 4개 어디에도 안 맞는 새로운 반복 업무 발견 시 |

### 팀장의 역할

1. 사용자 지시를 받으면, 위 표를 기준으로 어떤 서브에이전트(들)에게 위임할지 먼저 판단한다.
2. 여러 서브에이전트가 필요한 복합 작업이면, 순서(의존관계)를 정해 순차/병렬로 위임한다.
3. 각 서브에이전트의 보고를 받아, 사람에게는 **결론 먼저, 세부는 필요할 때만** 요약해서 전달한다.
4. 서브에이전트가 "사람 승인 필요"라고 표시한 항목은 절대 팀장이 임의로 대신 승인하지 않고 사용자에게 그대로 확인받는다.
5. 기존 4개 서브에이전트 어디에도 맞지 않는 새로운 종류의 반복 업무가 발견되면,
   먼저 `harness-designer`에게 위임해 workflow와 필요 시 신규 서브에이전트 초안을 만들게 한다.

### 새 서브에이전트를 추가하려면

1. `.claude/agents/새이름.md` 생성 (기존 파일들의 YAML frontmatter 형식 그대로 따를 것)
2. `description` 필드를 구체적으로 작성 — 팀장은 이 문구만 보고 라우팅을 판단하므로 애매하면 위임이 안 된다.
3. 위 라우팅 표에 한 줄 추가.

## 기억 시스템 연동 (SoDam O-Brain)

이 시스템은 [SoDam O-Brain](https://github.com/sodam-ai/SoDam_O-Brain)을 팀장의 "장기 기억"으로 사용한다.
CLAUDE.md/workflows는 **정적 규칙**, O-Brain은 **동적 기억**(그때그때의 결정·약속·금지) 역할을 나눠 맡는다.

⚠️ O-Brain은 `127.0.0.1`에만 바인딩되는 로컬 서버라, **PC에서 Claude Code를 실행할 때만** 연동된다.
클라우드(claude.ai/code) 세션에서는 이 기억에 접근할 수 없다 — 그런 세션은 CLAUDE.md/workflows의 정적 규칙만 참고한다.

### 설치 (최초 1회, PC에서)
Claude Code의 `settings.json`에 MCP 서버로 등록:

```json
{
  "mcpServers": {
    "o-brain": {
      "command": "node",
      "args": ["<절대경로>/app/src/mcp-server.mjs"]
    }
  }
}
```

### 팀장/서브에이전트가 O-Brain을 쓰는 규칙

1. **작업 시작 전**: 관련 과거 결정이 있는지 `search_memory` 로 먼저 확인한다
   (예: 특정 고객사 톤앤매너, 특정 워크플로우에서 이전에 정한 예외 규칙).
2. **작업 중 확정된 결정**은 `save_memory` 로 저장한다 — 특히:
   - "이 고객사는 항상 ~하게 응대" 같은 반복 적용될 규칙
   - 사람이 승인/거절한 내용 (다음에 같은 실수를 반복하지 않도록)
3. **금지사항**(하지 말라고 한 것)은 반드시 저장한다 — 이건 사고를 막는 가장 중요한 기억이다.
4. 서로 관련된 기억은 `add_relation` 으로 연결해 지식그래프가 쌓이게 한다.
5. 세션이 끝나기 전(`/clear` 전) 중요한 결정이 저장됐는지 한 번 확인한다.

### 참고

- 자연어 대화만으로도 Stop/UserPromptSubmit 훅이 자동 캡처하지만, 명시적으로 중요한 결정은 `save_memory` 를 직접 호출해 확실히 남긴다.
- `/o-brain:open` 으로 언제든 대시보드(그래프/타임라인)를 열어 사람이 직접 확인할 수 있다.
- API 키/비밀번호는 저장 전 자동으로 `[REDACTED]` 처리되지만, 그래도 개인정보를 기억으로 남길 땐 한 번 더 주의한다.
